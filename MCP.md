# MCP Server — Design Document

## 1. Vision

The current analysis pipeline produces static documentation (OpenAPI specs, SDL schemas) from captured traffic. This works well for REST APIs with clean URL patterns, but breaks down in practice:

- **Protocol diversity** — REST.li, JSON:API, persisted GraphQL, and proprietary RPC protocols all require specialized parsers. Each new protocol means new extraction steps, new assembly steps, new output formats. The combinatorial explosion is unsustainable.
- **Schema verbosity without actionability** — A 2000-line OpenAPI spec documents an API exhaustively but doesn't help anyone *call* it. Users still need to figure out auth, construct headers, manage tokens, and assemble request bodies by hand.
- **Response schemas are unnecessary for LLMs** — An LLM can interpret a raw JSON response directly. Generating and maintaining response schemas adds complexity without value when the consumer is an AI agent.

The MCP path takes a different approach: instead of producing documentation, it produces **executable tools**. Each business capability observed in the captures (search for a route, add a loyalty card, fetch account balance) becomes an MCP tool that Claude can call directly. The server constructs the HTTP request from a template, injects auth headers, makes the call, and returns the raw response.

This is protocol-agnostic by design. The tool definition stores a concrete request template — method, path, headers, body structure. Whether the underlying API is REST, REST.li, JSON:API, or a persisted GraphQL endpoint with opaque hashes, the template captures exactly how to call it. No protocol-specific parser needed.

### What changes

| Aspect | Current (analyze) | New (MCP) |
|--------|-------------------|-----------|
| Output | OpenAPI YAML, GraphQL SDL | Tool definitions + auth script |
| Consumer | Humans, Restish, code generators | Claude via MCP |
| Protocol support | REST + GraphQL (with dedicated parsers) | Any HTTP-based protocol |
| Response handling | Full schema inference | Raw response returned to LLM |
| Auth | Standalone script + Restish config | Integrated into server, auto-refresh |

The existing analyze pipeline remains available for users who want formal documentation. The MCP path is an alternative output target, not a replacement.

## 2. Extended storage layout

The managed storage directory (`~/.local/share/spectral/apps/<name>/`) gains new files alongside the existing captures:

```
~/.local/share/spectral/
└── apps/
    └── <name>/
        ├── app.json                # Extended with base_url
        ├── captures/               # Existing capture sessions (unchanged)
        │   └── ...
        ├── tools/                  # NEW — one file per MCP tool
        │   ├── search_routes.json
        │   ├── get_account.json
        │   └── add_loyalty_card.json
        ├── auth_acquire.py         # NEW — LLM-generated auth script
        └── token.json              # NEW — persisted auth state
```

| File | Role |
|------|------|
| `app.json` | Existing app metadata, extended with `base_url` |
| `captures/` | Existing capture sessions (unchanged) |
| `tools/<name>.json` | One file per business capability — description + request template |
| `auth_acquire.py` | Auth script generated by the LLM — handles credential prompts, login, token refresh |
| `token.json` | Current auth state — headers to inject, refresh token, expiration |

The `tools/` directory and `auth_acquire.py` are generated by a new analysis pipeline path (section 6). They can also be hand-authored or edited after generation.

## 3. File format specifications

### 3.1 `app.json` — extended

The existing `AppMeta` model gains one field:

| Field | Type | Description |
|-------|------|-------------|
| `name` | string | App slug (existing) |
| `display_name` | string | Human-readable name (existing) |
| `created_at` | string | ISO 8601 timestamp (existing) |
| `updated_at` | string | ISO 8601 timestamp (existing) |
| `base_url` | string, optional | API base URL (new) — e.g. `https://api.example.com/v2` |

The `base_url` is populated during analysis (reusing the existing `DetectBaseUrlStep`) and used by the MCP server to resolve relative paths in tool definitions.

### 3.2 `tools/<name>.json` — tool definition

Each file defines one MCP tool corresponding to a business capability observed in the captures.

#### Top-level fields

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `name` | string | yes | Tool identifier, snake_case (e.g. `search_routes`) |
| `description` | string | yes | What this tool does in business terms, for the LLM |
| `parameters` | JSON Schema object | yes | Standard JSON Schema describing the tool's input (same format MCP uses natively) |
| `request` | object | yes | HTTP request template |

#### `parameters` — JSON Schema

The `parameters` field is a standard JSON Schema `object` with `properties`, `required`, etc. This is the exact format MCP uses for tool input schemas, so no translation is needed when registering tools. It supports any JSON type: strings, numbers, booleans, arrays of objects, nested structures.

#### `request` — HTTP request template

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `method` | string | yes | HTTP method (`GET`, `POST`, `PUT`, `DELETE`, `PATCH`) |
| `path` | string | yes | Relative path from `base_url`, may contain `{param}` segments |
| `headers` | object | no | Fixed headers to send (keys are header names, values are literal strings) |
| `query` | object | no | Query parameters — values are either literal strings or `{"$param": "name"}` references |
| `body` | object or null | no | Request body template (see templating below) |
| `content_type` | string | no | Body content type. Defaults to `application/json` |

#### Templating system

The request template uses a minimal substitution system:

- **Fixed values** — Any literal value in the template is sent as-is. These are values observed in the captures that are always the same (API versions, fixed flags, content types).
- **Parameter references** — The marker `{"$param": "name"}` in any position (body field value, query parameter value, header value) is replaced at runtime with the LLM-provided parameter value. The `name` must match a property in `parameters`. Since parameters are JSON Schema typed, a `$param` marker can substitute any JSON value — a string, a number, or an entire array of objects. This avoids the need for loops or iteration in the template: if the API expects a variable-length list, the whole list is a single parameter.
- **Path parameters** — Standard `{param_name}` syntax in `request.path`. The `param_name` must match a property in `parameters`.

#### Example tool definition

```json
{
  "name": "search_routes",
  "description": "Search for train routes between two stations on a given date",
  "parameters": {
    "type": "object",
    "properties": {
      "origin": {
        "type": "string",
        "description": "Departure station name or code"
      },
      "destination": {
        "type": "string",
        "description": "Arrival station name or code"
      },
      "date": {
        "type": "string",
        "description": "Travel date in YYYY-MM-DD format"
      },
      "passengers": {
        "type": "array",
        "description": "List of passengers with type and count",
        "items": {
          "type": "object",
          "properties": {
            "count": {"type": "integer"},
            "type": {"type": "string", "enum": ["adult", "child", "senior"]}
          },
          "required": ["count", "type"]
        },
        "default": [{"count": 1, "type": "adult"}]
      }
    },
    "required": ["origin", "destination", "date"]
  },
  "request": {
    "method": "POST",
    "path": "/api/v1/search",
    "headers": {
      "X-Client-Version": "3.2.1"
    },
    "body": {
      "origin": {"$param": "origin"},
      "destination": {"$param": "destination"},
      "outward_date": {"$param": "date"},
      "passengers": {"$param": "passengers"},
      "currency": "EUR"
    }
  }
}
```

In this example, `currency` and `X-Client-Version` are fixed values observed in the captures. The `$param` markers are replaced at call time with values the LLM provides. The `passengers` parameter is an array of objects — the LLM constructs the full array and it's substituted as a whole, no loop needed in the template.

#### Form-encoded requests

When `content_type` is `application/x-www-form-urlencoded`, the `body` object is serialized as form fields instead of JSON. The same `$param` templating applies to field values.

### 3.3 `auth_acquire.py` — auth script

A Python module generated by the LLM that contains only the two auth functions — no framework code, no entry point. Spectral imports and calls these functions at runtime, providing all the surrounding infrastructure (credential prompting, token caching, expiry checking) from its own codebase. This means framework utilities can be improved across all apps when spectral is updated, without regenerating the auth scripts.

The module defines two functions:

| Function | Required | Purpose |
|----------|----------|---------|
| `acquire_token()` | yes | Full auth flow: prompts for credentials, returns auth state |
| `refresh_token(current_refresh_token)` | no | Token refresh: takes refresh token string, returns new auth state |

`acquire_token` takes no arguments. It prompts for credentials using helper functions injected by spectral at load time — this is necessary because multi-step auth flows (request OTP, wait for user input, verify code) cannot be pre-collected into a credentials dict upfront. The function manages the entire interactive flow internally.

Both functions return a dict with the following fields:

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `headers` | dict | yes | HTTP headers to inject into every request (e.g. `{"Authorization": "Bearer ey...", "Cookie": "session=abc"}`) |
| `refresh_token` | string | no | Refresh token for later use with `refresh_token()` |
| `expires_in` | integer | no | Token lifetime in seconds |

Key design decisions:

- **Functions only** — The file contains only `acquire_token()` and optionally `refresh_token()`. No framework code, no entry point, no caching logic. Spectral loads the module dynamically and calls the functions, wrapping them with its own token caching, expiry detection, and error handling. This keeps auth scripts small and lets framework improvements propagate without regeneration.
- **Headers, not tokens** — The functions return the actual HTTP headers to inject, not a raw token. This handles all auth mechanisms uniformly: Bearer tokens, cookies, API keys in custom headers, or any combination.
- **Credential prompting via injected utilities** — `acquire_token()` prompts for credentials (passwords, OTP codes) using helper functions that spectral injects into the module's namespace at load time. This lets spectral control how prompting works depending on the context (interactive terminal via `/dev/tty`, GUI callback, test harness, etc.) without changing the auth script.
- **Stdlib only** — The functions use only Python standard library (`urllib.request`, `json`, `base64`, etc.) to avoid dependency issues.

### 3.4 `token.json` — persisted auth state

Stores the current authentication state, written by the CLI after running the auth script:

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `headers` | object | yes | HTTP headers to inject (output of `acquire_token` or `refresh_token`) |
| `refresh_token` | string | no | For the refresh flow |
| `expires_at` | number | no | Epoch timestamp (seconds) when the token expires |
| `obtained_at` | number | yes | Epoch timestamp (seconds) when the token was obtained |

The MCP server reads `token.json` before each request to get the current auth headers. The separation between `auth_acquire.py` (how to get tokens) and `token.json` (current token state) means the potentially expensive/interactive auth flow only runs when needed.

## 4. MCP server behavior

### 4.1 Transport and startup

The `spectral mcp` command starts an MCP server on **stdio** transport. It scans all apps in managed storage for `tools/` directories and exposes every tool it finds.

### 4.2 Tool namespacing

Tools are namespaced by app to avoid collisions across apps:

| App | Tool file | MCP tool name |
|-----|-----------|---------------|
| `sncf` | `tools/search_routes.json` | `sncf_search_routes` |
| `sncf` | `tools/book_ticket.json` | `sncf_book_ticket` |
| `edf` | `tools/get_consumption.json` | `edf_get_consumption` |

The pattern is `<app_name>_<tool_name>`. The MCP tool description is taken directly from the tool definition file.

### 4.3 Request construction

When Claude calls a tool, the server:

1. Loads the tool definition from `tools/<name>.json`
2. Validates the provided parameters against the parameter schema
3. Resolves the full URL: `app.json:base_url` + `request.path` (with path parameter substitution)
4. Builds the request body by walking the body template and replacing `$param` markers
5. Merges fixed headers from the tool definition with auth headers from `token.json`
6. Executes the HTTP request
7. Returns the raw response body to the LLM (status code + headers + body)

The LLM interprets the response directly — no schema validation or transformation on the server side.

### 4.4 Auth cascade

Before making any request, the server checks auth state with the following cascade:

| Step | Condition | Action |
|------|-----------|--------|
| 1. Valid token | `token.json` exists and `expires_at` is in the future (or absent) | Use `headers` from `token.json` |
| 2. Auto-refresh | Token expired, `refresh_token` is present, `auth_acquire.py` defines `refresh_token()` | Run `refresh_token()`, update `token.json`, retry |
| 3. Auth required | No valid token and no refresh possible | Return an error instructing the user to run `spectral query login <app>` |

Auto-refresh is transparent to the LLM — the tool call succeeds as if the token were valid. Only when refresh fails or is unavailable does the server surface an error.

The server never runs `acquire_token()` itself because that function requires interactive credential input. The initial login is always user-initiated via the CLI.

## 5. CLI commands

### 5.1 `spectral mcp`

Starts the MCP server on stdio. This is the command users configure in their MCP client (Claude Desktop, Claude Code, etc.).

The server is a single process that exposes tools from all apps that have a `tools/` directory. It watches for changes to tool definitions and auth state at runtime — no restart needed when new tools are generated or tokens refreshed.

### 5.2 `spectral query login <app>`

Runs the interactive authentication flow for an app:

1. Dynamically imports `auth_acquire.py` from the app's storage directory
2. Calls `acquire_token()` — the function prompts for credentials and handles the full auth flow via `/dev/tty`
3. Writes the returned headers, refresh token, and computed `expires_at` to `token.json`

This is the only command that requires user interaction. It must be run before the MCP server can make authenticated requests for the app.

### 5.3 `spectral query refresh <app>`

Manually triggers a token refresh:

1. Loads `token.json` to get the current `refresh_token`
2. Loads `auth_acquire.py` and calls `refresh_token(current_refresh_token)`
3. Updates `token.json` with the new auth state

The MCP server auto-refreshes on demand, so this command isn't needed during normal use. But it's useful for keeping tokens warm in a crontab (e.g. `*/30 * * * * spectral query refresh sncf`) or for debugging auth issues when the server isn't running.

### 5.4 `spectral analyze <app> --mcp`

Runs the tool generation pipeline on an app's captures:

1. Loads and merges all captures for the app
2. Runs the MCP analysis pipeline (section 6)
3. Writes `tools/*.json`, `auth_acquire.py`, and updates `app.json` with `base_url`

This is the same `analyze` command as before, with `--mcp` selecting the MCP output path instead of OpenAPI/SDL. The existing flags still apply: `--model` to override the LLM model, `--debug` to save prompts, `--skip-enrich` to skip the LLM enrichment step. Without `--mcp`, the command produces OpenAPI/SDL as before.

## 6. Analysis pipeline (tool generation)

The MCP pipeline is fundamentally different from the existing REST/GraphQL pipeline. The existing pipeline is mostly mechanical (schema inference, extraction, assembly) with the LLM enriching the output. The MCP pipeline is LLM-driven: the LLM does the hard work (identify capabilities, understand what varies, build templates) and we assist it with mechanical investigation tools that surface the data it needs.

The pipeline has two LLM phases: first identify what tools to build (lightweight, sees summaries), then for each tool, investigate the traces and build the definition (detailed, with investigation tools). This avoids a single overwhelming prompt and lets each tool be built with focused context.

### 6.1 Preliminary steps

These reuse existing infrastructure to prepare the data:

| Step | Source | What it provides |
|------|--------|-----------------|
| Correlate UI ↔ API | `correlator.py` | Maps each API call to the user action that triggered it |
| Detect base URL | `steps/detect_base_url.py` | `base_url` for `app.json` and tool path resolution |
| Filter traces | `steps/filter_traces.py` | Keeps only business API traces |
| Analyze auth | `steps/analyze_auth.py` | Auth mechanism detection (`AuthInfo`) |

Auth analysis runs in parallel with the tool generation steps below.

### 6.2 Greedy extraction loop (step 1 → step 2, repeated)

An iterative loop that alternates between identifying a capability and building its full tool definition, one tool at a time. After each tool is built, its definition is used to mechanically remove matching traces from the pool before the next iteration.

**Each iteration has two phases:**

**Phase 1 — Identify one capability (LLM with tools).** The LLM receives:
- Compact summaries of all remaining correlations (UI action, page context, method/URL/status per trace — no bodies)
- The list of tools already built (name + description), so it avoids duplicates
- Access to investigation tools (section 6.3) to dig deeper when the summary is ambiguous

It produces one tool candidate: proposed `name`, `description`, and a list of `trace_ids`. Or a stop signal: nothing useful remains.

**Phase 2 — Build the tool definition (LLM with tools).** The LLM receives the tool candidate and uses investigation tools to explore the traces, understand the request structure, and produce the complete tool definition (section 3.2 format).

**Between iterations — mechanical trace cleanup.** The completed tool definition contains the request template with fixed values and `$param` markers. This template is used to match remaining traces in the pool: same method, URL matches the path pattern (with `{param}` as wildcards), and fixed values in the body match. Matching traces are removed from the pool — this is what ensures generalization. Even if the LLM only claimed 2 traces for "search routes", all 15 search traces get removed because they match the template.

**Stopping conditions:**
- The LLM explicitly says nothing useful remains in the pool
- The trace pool is empty
- A maximum iteration count is reached (safety net)

This interleaved approach solves the generalization problem naturally: the tool definition **is** the matching pattern, so we don't need to invent a separate discriminant extraction step. It also means each phase 1 sees a progressively cleaner trace pool.

### 6.3 Investigation tools

These tools are available in both phases of each iteration.

| Tool | Input | Output | Purpose |
|------|-------|--------|---------|
| `inspect_trace` | `trace_id` | Full trace: headers, decoded request body, decoded response body, status, timing | See the actual data in a specific request/response |
| `infer_request_schema` | `trace_ids[]` | Merged JSON Schema of request bodies with `observed` values per field | See which request fields exist across traces, their types, and which values vary vs stay the same |
| `query_traces` | `expression` (jq string) | jq output (JSON) | Run arbitrary jq queries against all traces — filter by URL/path/status, extract body fields, aggregate across traces. Each trace has fields: `id`, `method`, `url`, `path`, `status`, `request_headers`, `request_body`, `response_body`. Output capped at 8000 chars to force precise queries |
| `decode_base64` | base64 string | Decoded content | Existing tool — decode opaque values |
| `decode_url` | percent-encoded string | Decoded string | Existing tool — decode URL components |
| `decode_jwt` | JWT string | Header + payload | Existing tool — understand auth tokens |

The `infer_request_schema` tool is the key enabler. When the LLM gives it a set of trace IDs, the mechanical schema inference (reusing `schemas.py`) merges all request bodies and returns an annotated schema showing, for each field, the type, format, and up to 5 observed values. The LLM can then see at a glance:
- Fields where all observed values are identical → fixed values in the template
- Fields where values differ across traces → parameters
- Nested structures, arrays, enums — all inferred mechanically

This dramatically reduces what the LLM needs to figure out. Instead of reading N raw request bodies and mentally diffing them, it gets a pre-computed summary of what varies.

For responses, no schema inference — the LLM sees raw response bodies via `inspect_trace` when it needs to understand what an API returns. Response bodies are often large and deeply nested; a mechanically inferred schema would be noisy and unhelpful. The LLM reads actual examples and draws its own conclusions.

#### LLM output

The final tool definition (section 3.2 format): `name`, `description`, `parameters` (JSON Schema), `request` (template with `$param` markers).

### 6.4 Auth script generation

Reuses the existing `GenerateAuthScriptStep`, adapted: the generated `acquire_token` function must take no arguments (prompts via injected utilities) and return the new `headers`-based dict format (section 3.3) instead of the current `token`-based format. The framework code from `auth_framework.py` is no longer embedded in the script — spectral provides it at runtime. Token state is stored in the app's `token.json` in managed storage.

### 6.5 Pipeline flow

1. Load and merge captures for the app (existing)
2. Correlate UI actions with API traces (existing)
3. Detect base URL (existing)
4. Filter traces to business API (existing)
5. **In parallel:**
   - a. Analyze auth on all traces (existing)
   - b. **Greedy extraction loop** (section 6.2) — alternates identify → build → cleanup until the trace pool is exhausted
6. Generate auth script (existing, adapted) — depends on 5a
7. Write outputs — `app.json` (update `base_url`), `tools/*.json`, `auth_acquire.py`
